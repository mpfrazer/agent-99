## Design

### Interface

All backends share a common base class:

```python
class BaseMemory:
    def add(self, role: str, content: str) -> None: ...
    def history(self) -> list[dict]: ...   # [{role, content}, ...]
    def clear(self) -> None: ...
```

### Factory

```python
def create_memory(config: MemoryConfig) -> BaseMemory: ...
```

Called by the agent runner (cli.py ‚Üí loop.py) to instantiate the right backend.

### Backends

#### NoneMemory (type: "none")
- `add` and `clear` are no-ops.
- `history()` always returns `[]`.
- No storage, no imports beyond the base class.

#### SqliteMemory (type: "sqlite")
- Stores messages in a single `messages` table: `(id INTEGER PK, role TEXT, content TEXT, created_at TEXT)`.
- `path` from `MemoryConfig` is the DB file path; defaults to `":memory:"` if None (useful for tests).
- Uses stdlib `sqlite3` ‚Äî no extra dependencies.
- `add` inserts a row.
- `history()` returns all rows ordered by id as `[{role, content}]`.
- `clear()` deletes all rows.

#### ChromadbMemory (type: "chromadb")
- Scaffold only: raises `NotImplementedError` with a clear message.
- Keeps the slot open for a future implementation without breaking the factory.

### Integration with loop.py

The loop currently doesn't use memory ‚Äî that's intentional for this PR.
The memory interface is designed so that `loop.py` can later call:
  `memory.history()` to prepend past turns to messages, and
  `memory.add(role, content)` after each turn.

This wiring is left for a future step to keep this change focused.
